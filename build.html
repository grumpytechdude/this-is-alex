How I built this site

TL;DR - HTML and CSS. Go ahead - turn off Javascript!

Goals

I've worked on webapps my whole career. Sure, I got started as a teenager editing phpBB and using Dreamweaver, but once I got my degree and became a programmer, it's been webapps all the way. Not really intentionally - that's just how it panned out. And I'm glad! I really enjoy webapps.

However, I've also always worked on products somebody else had written. And they'd always used some framework that we had to keep using. First it was a homebrewed framework built over years before I turned up, next it was Sencha 5+6, next it was Sencha 4. I'm literally regressing!

To combat this, and bring some modernity to our Sencha 4 project, I started a new UI in React. We'll make new, dedicated UIs in React, taking a persona-focused approach. Instead of shoving everything onto a single page, now we're tailoring the experience so users don't see things they're not interested in.

And React is great! It's fun, it reminds me of the good ol' days of writing websites. But, you know, better.

But React is still JS. It has to download JS to render the page. Next.js is a great workaround for that, but (at work, at least) I don't fancy installing and maintaining a node runtime on our clients' servers.

So instead, I wondered how fast I could make this site. How easy. How few build steps. How much can I achieve, with such a little amount?

Method

The Content Itself

Plain old HTML, with plain old CSS! HTML5 and CSS3 have been around for years, so I thought I'd give them a whirl. I spun up IntelliJ IDEA and created a new HTML5 project. I added two files: index.html and index.css . Add in some <h1>'s and some <hr\>'s (who else had forgotten they existed?!) and we're good to go. Day one, let's get this online, no fuss.

Getting Online

I've used AWS, and I've used Netlify, and I've used Apache Server and Apache Tomcat and I want the simplest things. So I made a github repo, pushed my one commit, and cracked open netlify.

Four clicks. Once I'm signed in, it takes four clicks before they've hooked into my repo, downloaded the code, built the code (which I believe, in this instance, means "just load that whole directory there"), and given me an auto-generated url: https://frosty-ptolemy -5ac0b4.netlify.com/ You'll notice that's a https url too - not a http in sight!

They even give you this badge you can stick anywhere, and it'll tell you if your site is building:

After that, I pointed my alex-sinclair.dev domain at their name servers, they automatically provisioned a https certificate with Lets Encrypt, and now a site have I!

Statistics

Building a site with no JS is fun and all - but what's the point? Well, let's look at some pros!

  1. Four seconds between pushing code, and it being live.
  2. No build steps, no Jenkins, nothin'.
  3. 100/100 performance, accessibility, and best practice scores on lighthouse. Check it for yourself!
  4. This website is almost an exercise in minimalism. It's honestly a real treat.
  5. It's easy. I've not had to think about so few things at once, almost ever.
  6. You can learn how I've done it by looking at the source code. Most websites have a tangled mess of auto-generated code that it's essentially impossible to learn from.
  7. I'm helping to save the planet. Data transfers use electricity, and by not including tracking, or JS, or images, I keep my carbon footprint small. At the time of writing, index.html is 1.8KB, and index.css is 689B. That's 2500 bytes. I noticed Netlify was returning a 404 page for my favicon at 2KB, so by uploading a favicon I actually save bytes! Now it's 4 requests at 3.3KB transferred total.
    In Comparison, the google.com homepage makes 29 requests to load 439KB - to display 14 hyperlinks and a text box. And cnn.com makes 78 requests to load 2MB of data.

There are downsides, however. Plenty of them!

  1. webhint.io shows just how many things I'm not in control of. Cache headers and cross-origin mostly. Netlify serves my content from their CDN, so apparently css comes from cloudfront.net. If I were running my own server, I could setup all sorts of delightful headers.
  2. Duplication! With a build system we can inline css files, or auto-import them. Here, I have to setup the same header for each page. It would be easy to change a header in one place, and not another, and introduce bugs.
  3. No Tests. Not having a build means I can't run tests. I mean, I can - but I'd have to put them somewhere else. Or I could move this content to a /public source folder, and serve that instead.
  4. This is highly unnecessary! Gatsby.js combines your React with a CMS (or Markdown files) to render your site as static html. That aught to be just as fast as this little site.
  5. This site is dull. Boring. There's no interactivity; no flashing menus. I'd never advocate a site like this in a professional environment - but I think it's perfect for me, for now.
